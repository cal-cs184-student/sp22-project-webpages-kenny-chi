Assignment #4: Clothism - Kenny & Kevin Ch(a)i

This project involved using a mass and spring-based system to simulate a cloth. There were strings with different constraints, such as structural, shearing, and bending. We built data structures like a spatial map to define the cloth. We also implemented collisions with spheres, planes, and the cloth itself (self-collisions). Finally, we created shaders to change the look of the cloth as it moved with gravity and collided using GLSL.

Part 1: Masses and springs

In this part, we created a system of point masses and springs. We populated the cloth’s point_masses and springs vectors. It was important to check whether the orientation of the cloth was HORIZONTAL or VERTICAL while creating this system. When creating the springs, we had to do some index arithmethic to determine what type of constraint to use.

In order to implement the masses and springs, we looped through num_height_points by num_width_points and evenly spaced out points along each axis. In the vertical orientation z is a random small float.

Part 2: Simulation via numerical integration

In this part, we implemented the spring masses of the cloth responding to graviational forces. We did this by summing all of the external forces of the point masses together. Then, we used Verlet integration to compute new positions for each point mass as they were being accelarated by the force we calculated previously. The last thing that we added was a constraint to the changes in distance. If the distance was more than 1.1x the springs rest length, we halved distance in each direction to prevent spring deformation.

Changing the Spring Constant

Changing the spring constant results in a cloth that is either more or less stiff. When the spring constant is smaller, there are more folds in the cloth as it is much more flexible. When the spring constant is increased, the cloth was much stiffer and fell down more uniformly.

The density of the cloth affected the gravity on the cloth because the effect of gravity on the point masses was greater with the greater density, which created more wrinkles in the cloth. Changing damping

Spring damping refers to the decay in oscillations in the spring. In our cloths, we could see a clear different when the damping was set to zero (all of the energy was conserved) versus when we set the damping to one percent. Not only did the cloth move much slower with more damping, it also moved much less due to the reduced oscillations of the spring. We can clearly see the difference after the cloth has hit the bottom and the springs are oscillating.

Part 3: Handling collisions with other objects

In this part, we handled our cloth colliding with different objects like a sphere or a plane.

To handle collisions with a sphere, we first checked if a point mass intersects with or inside of a sphere. If it does intersect, then we bump it up to the surface of the sphere. We first calculate the tangent point. Then we calculate the correction vector in order to reach the tangent point. Finally, we update the point mass’s position to be the old position adjusted by the correction vector scaled down by friction.

To handle collisions with phone, we followed the exact same logic as above. There were a little more complexities like calculating the tangent point. We had to subtract SURFACE_OFFSET to account for a small displacement.

Part 4: Handling self-collisions

In this part, we handled the cloth colliding with itself. Before implementing part 4, the cloth behavior was strange and would sometimes clip through. To handle self-collisions, we implemented hash_position, build_spatial_map, and self_collide.

hash_position takes in a mass position and partitions that space into 3D boxes. Using the position, we truncated those coordinates using floor to the nearest 3D box. Then we created a unique number out of those three coordinates to represent a unique key in our hash table.

build_spatial_map loops over all the point masses and populates a hash map using hash_position.

self_collide takes in a point mass and looks at potential candidates for collision. For each candidate, we checked if they were 2 * thickness apart. If they were, then we computed a correction vector to apply to the position. The final correction vector is the average of all the computed correction vector divded by simulation_steps. We then applied the final correction vector to our point mass position.

Part 5: Shaders

Part 5 involved us creating our own shaders using GLSL. We implemented different types of shading including diffuse shading, blinn-phong shading, texture mapping, displacement and bump mapping, and environment-mapped reflections.

Shader Program

A shader program is a program that tells the computer how to render each pixel at each 3D location. Vertex shaders transform the vertices into coordinates on the screen and fragment shaders compute attributes such as z depth and the shape’s color after rasterization.

In other words, a vertex shader takes care of the vertex interpolation of the polygons, and the fragment shader takes care of the pixels between the vertices. The way they interact is that the fragment shader comes after rasterization and then vertex shader.

Blinn-Phong shading model

The Blinn-Phong shading model breaks down how a model should look based on the ambient, diffuse and specular light. This is shown in our code through term1, term2, and term3. The specular part is reflected light, diffuse is the light bounding in all directions, and the ambient light is the light that fills the object.
